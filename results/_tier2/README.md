# `results/_tier2/` — Circularity-control reruns (Tier 2)

**What's in this folder:** Replication runs designed to defuse a circularity concern: the original batteries were written by Haiku, which was also a response model and judge. If Haiku is systematically biased toward its own phrasing, the spec advantage could be an artifact. Tier 2 tests for that.

## What Tier 2 tests

For three subjects spanning the gradient (Ebers, Yung Wing, Zitkala-Sa), the test battery is regenerated by a different model (GPT-5.4) and the response is generated by a non-Haiku model (Sonnet or Gemini Pro). If the spec advantage persists, circularity is not the cause. The paper reports 5 of 6 cells positive in this replication.

## Contents

Three per-subject subfolders:

- `global_ebers/`, `global_yung_wing/`, `global_zitkala_sa/`: Tier 2 data per subject.

Each subfolder contains:

- `tier2_<response_model>_results.json`: Response model's answers to the GPT-5.4-generated battery.
- `tier2_<response_model>_judgments_<judge>.json`: Each judge's scores of those responses.
- `.rl_backup` files: Pre-rerun backups; not primary data.

`<response_model>` here is `sonnet` or `gemini_pro`. `<judge>` is one of the seven short judge names.

## How naming works here

Same short-name conventions as everywhere else (`docs/FILE_NAMING.md`). The `tier2_` prefix distinguishes these runs from the standard per-subject runs. Files live under `global_<subject>/` to match the subject-folder layout in `results/`.

## Where these files come from / go to

Inputs: GPT-5.4-generated battery (which replaces the standard Haiku-generated battery for these cells). Outputs feed paper §6 (content-specificity / circularity section) and `docs/research/tier2_recompute_s114.json`.

## Caveats worth knowing

- Tier 2 covers three subjects, not all 14. The paper flags this explicitly as a replication subset, not a full rerun.
- `.rl_backup` files are rate-limit-retry backups saved mid-run. Treat the non-backup version as authoritative.
