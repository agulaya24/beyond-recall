# `results/hamerton/` — Results for Philip Gilbert Hamerton (primary subject)

**What's in this folder:** Every AI response and every judge score for Hamerton, the study's primary subject. Hamerton has the deepest set of conditions because he was the first subject piloted.

## Why Hamerton gets the richest data

Hamerton has the lowest C5 baseline of any subject (1.25 on the original 7-judge panel). Meaning: the model knows almost nothing about him from pretraining. That makes him the cleanest test of whether a behavioral specification can move the needle. Most method development and extra analyses (Letta stateful-agent test, full-stack spec refresh, etc.) happened on Hamerton first.

## Contents

All files follow the schema documented in `../README.md` and `docs/FILE_NAMING.md`. The shape for Hamerton:

- `results.json`, `results_v2.json`, `results_harmonized.json`: Model responses across the core conditions. `_v2` is the version the paper uses; `_harmonized` aligns battery format with the 13 global subjects.
- `judgments.json`, `judgments_harmonized.json`: Judge scores across core conditions.
- `manifest.json`: SDK versions, timestamps, models used.
- `<system>_results.json`, `<system>_retrieval.json`, `<system>_ingestion.json`: Per-memory-system controlled run (Mem0, Letta, Supermemory, Zep).
- `<system>_fullpipeline_*.json`: Per-memory-system native run (each system's own extraction).
- `<system>_judgments_<judge>.json`: One judge's scores for that system.
- `<system>_judgments_merged.json`: All judges merged.
- `baselayer_*.json`: Base Layer substrate (MiniLM + ChromaDB). Not a memory system; the spec layer.
- `c8_c9_*.json`: Raw-corpus condition (C8) and raw-corpus-plus-spec condition (C9).
- `fullstack_haiku.json`: The full-stack spec run with Haiku as the response model (used for the S113 refresh).
- `sonnet_judgments.json`, `opus_judgments.json`, `gpt4o_judgments.json`: The S114 backfill. These three judges were not in the original Hamerton judge panel for spec conditions, so S114 ran them to bring Hamerton onto the 5-judge primary panel.
- `gemini_pro_judgments.json`, `gpt54_judgments.json`: Additional per-judge scores.

## How naming works here

Standard per-subject schema from `docs/FILE_NAMING.md`. Judges use short names (haiku, sonnet, opus, gpt4o, gpt54, gemini_flash, gemini_pro). Systems use short names (mem0, letta, supermemory, zep, baselayer).

## Where these files come from / go to

Generated by scripts in `scripts/` (for example, `run_full_study.py`, `run_full_spec_rerun.py`, `judge_hamerton_5judge.py`). Consumed by the aggregate in `results/RESULTS_S113.json` and by the 5-judge primary recompute.

## Caveats worth knowing

- Hamerton's spec-condition judgments were originally scored by Haiku + Gemini Flash + GPT-5.4 + Gemini Pro only. S114 backfilled Sonnet, Opus, and GPT-4o so the 5-judge primary panel is complete.
- `judgments_harmonized.json` and `results_harmonized.json` are the cross-subject-comparable versions; they use the same battery format as the 13 globals.
